Every Cypher market moves through a defined sequence of states ŌĆö from the moment a creator locks in a bond to the moment winners claim their USDC. Understanding the lifecycle helps you know exactly what actions are available at each stage, who triggers them, and what to do if something goes wrong. The SDK's `marketPhase()` function gives you a real-time label for any market so your UI always reflects the correct state.

## Full lifecycle walkthrough

<Steps>
  <Step title="betting ŌĆö Market is open">
    **Who triggers it:** The market creator, by calling `createMarketAction` (or `createMarketMultiAction`).

    The creator posts a $10 USDC bond, sets the question, oracle type, lock timestamp, and resolve deadline. Pools are initialized. The market enters the `betting` phase immediately.

    During `betting`, anyone can call `placeBet` (or `placeBetAccuracy`). Each bet encrypts the prediction client-side and creates a `Position` account on Solana storing the ciphertext. The stake amount is transferred to the pool vault.

    This phase lasts until `lock_timestamp` is reached.
  </Step>

  <Step title="awaitingResolve ŌĆö Betting closed, resolution pending">
    **Who triggers it:** Automatic ŌĆö `lock_timestamp` has passed.

    No new bets can be placed. Anyone can call `lock_market` to transition the `MarketGroup` to `Locked` status, which permanently closes the pools. The oracle (Pyth feed or market creator) must now post the resolution before `resolve_deadline`.

    The market stays in `awaitingResolve` until either the oracle posts a result or the deadline passes.
  </Step>

  <Step title="pendingResolution ŌĆö Challenge window active">
    **Who triggers it:** The oracle (Pyth or market creator), by calling `postResolution`.

    The oracle posts a `ResolvedValue` (yes/no boolean, outcome index, or numeric value). A **dispute window** opens immediately and lasts between 24 and 48 hours depending on how the market was configured.

    During this window, **anyone** can call `flagResolution` to dispute the posted outcome. If no one disputes, the market proceeds to `awaitingFinalize`. If the market is flagged, it enters `disputed`.

    <Note>
      The 24ŌĆō48 hour challenge window is your community's safety net. If a market creator posts a wrong or manipulated outcome, any participant (or external observer) can flag it before payouts are distributed. You do not need to hold a position to flag a resolution.
    </Note>
  </Step>

  <Step title="awaitingFinalize ŌĆö Challenge window elapsed, awaiting finalize call">
    **Who triggers it:** Automatic ŌĆö challenge window has elapsed with no dispute.

    The resolution is considered accepted. Anyone can now call `finalizeResolution` to move the market forward. This is a permissionless call ŌĆö no special role required.

    Once `finalizeResolution` is called, the backend queues the Arcium MPC settlement computation.
  </Step>

  <Step title="MPC settlement ŌĆö Arcium computes winner mask (~10 seconds)">
    **Who triggers it:** The Cypher backend, after `finalizeResolution`.

    Once finalized, the Cypher backend submits all encrypted position payloads to the Arcium MXE cluster. The MPC circuit runs entirely inside an encrypted context ŌĆö no single party ever sees individual predictions in plaintext. The cluster determines winners and fires a callback to the Solana program.

    <Note>
      Arcium MPC settlement typically completes in **~10 seconds** on devnet. You can poll for settlement completion using `awaitComputation(client, computationOffset)`.
    </Note>

    The callback writes each position's `payout` field and marks the pool as `Settled`. Fees are transferred to the treasury and creator at this point.
  </Step>

  <Step title="claimable ŌĆö Winners can claim payouts">
    **Who triggers it:** MPC settlement callback completing successfully.

    The `MarketGroup` transitions to `GroupStatus::Settled`. Any participant with `position.payout > 0` can now call `claimPayout`. The USDC is transferred from the pool vault to the winner's wallet. The position is marked `Claimed`.

    The creator's $10 USDC bond is returned automatically if the resolution was accurate and undisputed.
  </Step>
</Steps>

## Exception states

These states are reached when something deviates from the normal path.

<CardGroup cols={2}>
  <Card title="disputed" icon="flag">
    Someone called `flagResolution` during the challenge window. The market is paused while a Cypher admin reviews the posted outcome. The admin can either confirm the original resolution or override it with a corrected outcome using `adminOverrideResolution`. If the dispute is upheld, the creator's bond is slashed.
  </Card>
  <Card title="refundable" icon="rotate-left">
    The market passed its `resolve_deadline` with no oracle resolution. All participants can call `claimRefund` to recover their original stakes. The creator's bond is slashed and sent to the treasury.
  </Card>
  <Card title="expired" icon="clock">
    All deadlines ŌĆö claim deadline and refund deadline ŌĆö have elapsed. Remaining vault funds can only be recovered by the platform admin. Participants should always claim before the claim deadline.
  </Card>
  <Card title="cancelled" icon="xmark">
    The market creator called `cancelMarket` **before any bets were placed**. Once a single bet exists, cancellation is blocked. The creator's bond is returned in full on a clean cancel.
  </Card>
</CardGroup>

## `marketPhase()` reference

The `marketPhase(market, nowSec?)` function from `@cypher-zk/sdk` computes the current lifecycle label from on-chain account data. Use it to drive UI state ŌĆö which buttons to show, what copy to display, and what actions to enable.

```ts
import { marketPhase } from "@cypher-zk/sdk";

const phase = marketPhase(market); // uses Date.now() / 1000 internally

switch (phase) {
  case "betting":
    // show "Place Bet" button
    break;
  case "awaitingResolve":
    // show "Awaiting Oracle" banner (resolver-only action available)
    break;
  case "pendingResolution":
    // show challenge window countdown + "Flag Resolution" button for anyone
    break;
  case "awaitingFinalize":
    // show "Finalize Resolution" button (permissionless)
    break;
  case "disputed":
    // show "Under Admin Review" banner
    break;
  case "claimable":
    // show "Claim Payout" button for winners
    break;
  case "refundable":
    // show "Claim Refund" button for all participants
    break;
  case "expired":
    // show "Market Expired" message ŌĆö no user actions available
    break;
  case "cancelled":
    // show "Market Cancelled" message
    break;
}
```

### All `MarketPhase` values

| Value | Condition | Clickable actions |
|---|---|---|
| `"betting"` | Market open, before `lock_timestamp` | `placeBet` |
| `"awaitingResolve"` | Locked, before `resolve_deadline` | `resolveMarket` (resolver only) |
| `"pendingResolution"` | Resolution posted, challenge window active | `flagResolution` (anyone) |
| `"awaitingFinalize"` | Challenge window elapsed, not disputed | `finalizeResolution` (anyone) |
| `"disputed"` | Resolution flagged | `adminOverrideResolution` (admin only) |
| `"claimable"` | Settled, before `claimDeadline` | `claimPayout` |
| `"refundable"` | Resolve deadline missed, before `refundDeadline` | `claimRefund` |
| `"expired"` | All deadlines elapsed | None (admin only) |
| `"cancelled"` | Creator cancelled before any bets | None |
